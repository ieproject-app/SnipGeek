Next.js has revolutionized React development by providing a robust framework that handles everything from server-side rendering to static site generation. In this post, we'll dissect its architecture.

## The App Router: A New Paradigm

The most significant change in recent versions of Next.js is the introduction of the App Router. It leverages React Server Components, allowing developers to build complex applications with less client-side JavaScript.

Here's a long code block to demonstrate syntax highlighting and scrolling. Notice the dark background and the horizontal scrollbar on smaller screens.

```tsx
// src/app/layout.tsx
import { Inter } from 'next/font/google';
import './globals.css';
import { ThemeProvider } from '@/components/theme-provider';
import { Analytics } from '@vercel/analytics/react';

const inter = Inter({ subsets: ['latin'] });

export const metadata = {
  title: 'My Awesome App',
  description: 'Generated by create next app',
};

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en" suppressHydrationWarning>
      <body className={inter.className}>
        <ThemeProvider attribute="class" defaultTheme="system" enableSystem>
          {children}
          <Analytics />
        </ThemeProvider>
      </body>
    </html>
  );
}
```

## Server vs. Client Components

Understanding the distinction is key:
- **Server Components**: Run on the server, can access databases and files, and are not interactive.
- **Client Components**: Run on the client, are interactive, and use hooks like `useState` and `useEffect`.

<Warning>
  Be careful when fetching data in Client Components. To avoid exposing sensitive keys, you should always call an API route or a Server Action instead of fetching directly.
</Warning>
