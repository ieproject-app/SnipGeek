To ensure clarity and prevent details from being missed (like publication status or syntax issues), we've agreed on a standard template for submitting requests. Using this format will help streamline our workflow and ensure all necessary information is provided upfront.

### Standard Prompt Template

**1. Clear & Concise Goal:**
Start with a single sentence that summarizes the main objective.
*Example: "Add a 'Reading List' feature to the application."*

**2. Detailed Breakdown & User Stories:**
Describe the feature or change in detail. Explain how a user would interact with it.
*Example: "A user should be able to click a bookmark icon on an article. This saves the article. The user can then view their saved articles from a list in the header."*

**3. Specific Technical & UI Requirements (If any):**
Mention any specific technologies, components, or UI behaviors you have in mind.
*Example: "Please use `localStorage` for storage. The list in the header should be integrated with the main header component."*

**4. Acceptance Criteria:**
List the conditions that must be met for the task to be considered complete.
*Example: "The bookmark icon changes state when clicked. The count in the header updates. The list displays correctly."*

**5. Things to AVOID:**
If there are any anti-patterns or specific things not to do, list them here.
*Example: "Do not use a separate page for the reading list; it must be integrated into the header."*

By adhering to this format, we can minimize back-and-forth and build features more accurately and efficiently.
