We recently ran into a classic front-end development puzzle. Our code blocks, which are rendered from MDX files using the fantastic Shiki syntax highlighter, just wouldn't accept our custom styles.

The goal was simple: give them rounded corners and some internal padding so the code doesn't sit flush against the edges of its container.

### The First Attempt: Global CSS

My first instinct was to add a global CSS rule in `src/app/globals.css`:

```css
pre.shiki {
  @apply rounded-lg p-6 my-6 overflow-x-auto;
}
```

In theory, this should have worked. We were targeting all `<pre>` elements with the class `shiki` (which is added by the highlighter) and applying our desired Tailwind classes. But... nothing happened. The code blocks remained stubbornly square and without padding.

### The Analysis

The problem likely came down to CSS specificity or the complex interaction between Next.js's build process, Tailwind's class injection, and Shiki's dynamic HTML generation. Our global style was probably being overridden or just wasn't being applied with enough "force" to the final rendered element.

### The Solution: Direct Component Rendering

Instead of relying on a global style to "find" the code blocks, we decided to take a more direct approach by telling MDX *exactly* how to render them.

This was done in `src/components/mdx-components.tsx`. This file acts as a dictionary for our MDX renderer.

We created a custom React component for the `<pre>` element:

```tsx
const MdxPre = ({ className, ...props }: React.HTMLAttributes<HTMLPreElement>) => (
    <pre
        className={cn(
            "rounded-lg p-6 my-6 overflow-x-auto",
            className
        )}
        {...props}
    />
);
```

And then we registered it:

```tsx
export const mdxComponents: MDXComponents = {
    // ... other components
    pre: MdxPre, // <-- This is the key part
}
```

By doing this, we're not leaving styling to chance. We're explicitly instructing the renderer: "Every time you encounter a `<pre>` block, use this component, which is guaranteed to have the classes for rounded corners and padding."

And it worked perfectly! It's a great reminder that sometimes a more direct, component-based approach is more robust than relying on global CSS selectors, especially in a modern component-based framework.
