While building this blog, we ran into a rather annoying visual issue: all `<code>` elements looked the same. This meant that short snippets of code within a paragraph (inline code) looked exactly like multi-line code blocks, which were supposed to have their own distinct background and padding.

### The Root Problem

By default, we applied a base style to the `<code>` tag in `globals.css`:

```css
code {
  @apply font-code relative rounded bg-muted px-[0.4rem] py-[0.2rem] font-mono text-sm font-semibold;
}
```

The problem is that our syntax highlighting library (Shiki) also generates a `<pre><code>...</code></pre>` HTML structure for code blocks. As a result, the `code` style above was also being applied to large code blocks, overriding the beautiful styling from Shiki.

### The Solution: Using `@layer` and a CSS Reset

The solution was to differentiate the CSS targeting. We kept the base style for `<code>`, but added a specific override for any `code` element found inside a `<pre>` tag.

We used Tailwind CSS's `@layer components` to define this custom style and added a reset rule:

```css
@tailwind base;
@tailwind components;
@tailwind utilities;

@layer components {
  /* Default style for inline code */
  code {
    @apply font-code relative rounded bg-muted px-[0.4rem] py-[0.2rem] font-mono text-sm font-semibold text-foreground/80;
  }

  /* Reset styles specifically for Shiki's code blocks */
  pre code {
    @apply bg-transparent text-inherit p-0 font-normal rounded-none static;
  }
}
```

With the `pre code` rule, we effectively "erase" the custom styles from inline code whenever it's inside a `<pre>` block, allowing Shiki's styles to take full precedence. As a result, inline code and code blocks now have distinct, appropriate appearances.
